## Context

This is **"Pensar es olvidar"**, a literary AI artifact for Google I/O 2026's "big number 1–10" challenge. It's a three-act web experience inspired by Borges's *Funes el memorioso* — a user spots digits hidden in a reel of scenes, two memory systems (Funes/verbatim, MIRAS/abstracted) ingest the events in parallel, and an automated auditor judges them on understanding. The thesis is that thinking requires forgetting.

The full spec lives in `pensar-es-olvidar-spec.md`. The code in this repo is **V0**, generated by Google AI Studio's Build mode from a subset of that spec. V0 is roughly 70% there. Your job is to take it from V0 to V1 ship-ready, and then add V1.5 (voice auditor) if time permits.

The author is Carlos Crespo Macaya. The artifact is for a Google I/O submission and is also a portfolio piece aimed at a top-tier applied AI engineering role. **Engineering legibility matters as much as the artistic surface.** Anyone who reads this repo should see a senior engineer's discipline: clean commits, real tests, observability, type safety, accessible code. No vibes commits.

### 2. Establish engineering hygiene (one focused session)

Before any feature work, set up the rails so every subsequent change is protected:

- Add ESLint (TypeScript strict + React hooks rules). Fix or suppress all existing violations in V0.
- Add Vitest. Write the first unit test against the Funes reducer (it's a pure function, easiest target). The test asserts: appending an event preserves all prior events verbatim, no abstraction occurs, ordering is correct.
- Add Playwright. Write one happy-path E2E test that opens the prologue, clicks Begin, and asserts the reel mounts. Just one — more come later.
- Add Promptfoo. Write the first eval against the MIRAS retention prompt: given a fixture of 12 events, the abstract must stay under 280 tokens and the latent theme word must surface by event 8. Use the spec's MIRAS prompt verbatim.
- Set up GitHub Actions: typecheck → lint → unit → integration → llm-eval → build. Branch protection on `main`. Block direct pushes.
- Add `.editorconfig`, `prettier`, and `lint-staged` with a `pre-commit` hook so commits are clean automatically.
- Commit each step separately with a clean message. The commit history is part of the portfolio.

### 3. Fix Soul issues first, then Structure, then Surface

This order is non-negotiable. The MIRAS prompt is the artifact's voice — every visual fix is meaningless if the prompt is generic.

For each fix, the loop is: write a failing test (or eval) → make the change → confirm green → commit. Even tiny visual changes get a Playwright visual-regression snapshot. This is what separates this repo from a vibe-coded weekend toy.

### 4. Specific things to verify or fix early

These are likely V0 issues based on how Build mode tends to behave; verify each and fix as needed:

- HTML `<html lang="...">` should update reactively when the user changes language (most likely V0 hardcoded `lang="en"`)
- Verify the Gemini prompts match the spec exactly. If Build mode paraphrased them — and it almost certainly did for at least one of the three (MIRAS, auditor, judge) — restore the spec versions verbatim. Externalize all prompts to `src/prompts.ts` if they aren't already, and make them per-language.
- The trial's auditor calls must be parallelized within each round (Funes query + MIRAS query running concurrently, not sequential), and the four rounds run sequentially. Verify the Promise structure.
- The MIRAS update on each correct click should not block the bloom animation. The bloom fires immediately on click; the Gemini call resolves async and updates the rail when ready.

- The bloom needs to feel ecstatic, not informational. 350–400ms warm-white flash with a slight scale animation, GPU-accelerated transform/opacity only. Audio chime via Web Audio API, pitched by digit (C4 through E5).
- Self-promotion footer: bottom-right, 12px, 40% opacity, hover to 100%, with a link to carloscrespomacaya.com. Hidden during bloom and trial.

## Project conventions

Create `CLAUDE.md` at the repo root with these rules baked in so future Claude Code sessions are consistent:

- TypeScript strict mode. No `any` without a `// reason:` comment.
- All Gemini prompts live in `src/prompts.ts`, externalized per language. Never inline a prompt in a component.
- All Gemini calls go through `src/lib/gemini.ts` which wraps the SDK with logging hooks. Never call the SDK directly from a component.
- All animations use `transform` / `opacity` only. No layout-thrashing properties.
- All user-facing text goes through i18next, including new strings. No hardcoded English in components.
- Color tokens live in Tailwind config (`funes-red`, `miras-blue`, `bloom`). Never use raw hex in components.
- Each PR includes: code changes, test changes, and (if it touches a prompt) Promptfoo eval changes. No exceptions.
- Commits use conventional commit format: `feat:`, `fix:`, `chore:`, `docs:`, `test:`, `refactor:`. Subject line ≤ 72 chars.
- Branch names: `feat/<short-name>`, `fix/<short-name>`. PRs squash-merge to `main`.

## When you're stuck

If something is genuinely ambiguous between the spec and the V0 code, follow this order:
1. Re-read the relevant spec section
2. Re-read this handoff doc
3. Ask Carlos before guessing

If a Gemini prompt is producing output that doesn't feel right (MIRAS abstracts that summarize instead of abstract, auditor responses that ignore the "be faithful to your nature" instruction), don't try to patch it with one-off tweaks. Iterate against the Promptfoo fixture until the contract holds, then commit the winning prompt.

If you find yourself wanting to make a change that "feels right" without a test or eval to back it up, stop. The whole point of this engineering setup is that intuition gets verified, not deployed.
